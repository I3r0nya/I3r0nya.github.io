UPX加壳脱壳及原理分析

加壳​

Windows下有专门的upx软件加壳，Linux下直接安装upx的apt包就能加壳脱壳

脱壳​

对于一般的upx加壳直接用upx -d就能脱，但是很多时候会对壳进行变种不得不手脱

32位​

32位的程序一般用ESP定律就能快速脱壳

ESP定律：本质是栈平衡，压缩壳（或者加密壳）一般在进入解压过程之前会pushad保护现场，在进行完pushad之后对esp下一个硬件断点，当壳的解压代码完成会popad恢复现场之后再跳转到程序真正的OEP，这时候就会在断点处停下，也就找到了原程序的OEP

用x32dbg打开upx加壳之后的notepad.exe

可以看到入口第一步就是pushad，这是壳的特征

如果x32dbg打开第一个指令不是pushad，有可能是tlscallback被调试拦截了，需要自己下断点到pushad处，或者用IDA动调

F8步过之后在ESP=0x000DFF54的地址打一个硬件断点

软件断点是用int 3实现的，本质是异常处理，只有在断点地址可写的时候才能使用；而硬件断点是cpu提供的，数量有限，一般是4个

F9让程序跑到断点位置停下

可以看到程序停在了popad之后一个指令

之前说过压缩壳的程序在解压完所有的代码之后就会跳转到程序真正的OEP位置，所以可以看到最后一条指令jmp notepad_upx.100739D，程序真正的OEP就是0x100739D，和未加壳的原程序对比一下，发现确实是这样

找到OEP之后就可以直接用工具脱壳了，x32dbg自带插件Scylla

点击IAT Autosearch可以自动找到IAT地址，然后点击Get Imports获取导入的dll，然后点击Dump就能dump下来脱壳的程序，最后点击Fix Dump重建dump下来的文件的IAT就完成了脱壳的过程

64位​

64位因为在调用函数的时候和32位不一样，所以先得单步调到寄存器入栈的地方再用硬件断点，或者单步跟踪法调到OEP的位置

单步跟踪法：在程序运行的时候，只运行向下跳的，向上跳的用F4跳过，直到看到跨节区的跳转（OEP）

因为开启了ASLR，所以可能地址不太一样，具体操作和32位一致，push完了之后在RSP的地址上打一个硬件断点，F9运行到pop的位置，下面的一个大跳转就是OEP的地址
