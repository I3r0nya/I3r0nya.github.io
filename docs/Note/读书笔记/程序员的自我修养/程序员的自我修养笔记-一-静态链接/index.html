程序员的自我修养笔记(一) | 静态链接

程序编译过程​

通常编译一个.c文件会使用gcc来编译，比如gcc test.c -o test

先写一个HelloWorld

#include <stdio.h>

int main() {
    printf("Hello World!\n");
    return 0;
}

在编译的过程中包含了四个过程: 预编译(Propress)、编译(Compile)、汇编(Assembly)、链接(Link)

预编译​

相当于 gcc -E test.c -o test.i 或者是 cpp test.c > test.i

预编译过程主要处理代码里面以#开始的预编译指令，比如展开宏定义、插入#include代码、删除注释、添加行号等

注意，如果代码中包含#pragma编译器指令，在这个过程中是保留的

编译​

相当于 gcc -S test.i -o test.S

编译过程是把预处理完后的文件进行词法分析、语法分析、语义分析以及优化，并产生汇编代码

汇编​

相当于 gcc -c test.S -o test.o 或者 as test.S -o test.o

汇编过程把汇编文件转换成机器可以执行的指令

链接​

汇编过程之后生成的test.o文件不能直接执行，需要执行链接过程来确定函数地址. 比如test.c中使用了printf()，但test.c里并没有定义printf()，这个函数是在标准库里面定义的，链接的一项任务就是重定位函数地址

链接过程主要包括地址和空间的分配、符号决议、重定位

目标文件格式​

ELF文件类型说明实例
Relocatable File包含代码和数据，可以用来链接成可执行文件或共享目标文件.o(Linux)/.obj(Windows)
Executable File包含了可以直接执行的程序elf(Linux)/.exe(Windows)
Shared Object File包含了代码和数据，有两种情况: 1.链接器使用这种文件和其他的Relocatable File和Shared Object File链接，产生新的目标文件 2.动态链接器可以将几个Shared Object File与Executable File结合，作为进程映像的一部分运行.so(Linux)/.dll(Windows)
Core Dump File当进程意外终止时，系统将进程的地址空间的内容及终止时的一些信息转储到Core Dump Filecore dump(Linux)

Magic最开始4个字节是ELF文件标识码：7F 45 4C 46. 7F对应ASCII字符中的DEL控制符，45 4C 46对应ELF，这四个字节称为ELF魔数

第5个字节是标识ELF文件类的，01表示ELF32，02表示ELF64

第6个字节是字节序，规定ELF文件时大端序还是小端序列，01表示小端，02表示大端

第7个字节规定ELF文件的主版本号，一般都是01

后面9个字节没有标准定义，有些平台会使用这9个字节作为扩展标志

Section Header Table​

Section Header Table位于ELF文件的尾部

/* Section header.  */

typedef struct
{
  Elf32_Word    sh_name;        /* Section name (string tbl index) */
  Elf32_Word    sh_type;        /* Section type */
  Elf32_Word    sh_flags;       /* Section flags */
  Elf32_Addr    sh_addr;        /* Section virtual addr at execution */
  Elf32_Off sh_offset;      /* Section file offset */
  Elf32_Word    sh_size;        /* Section size in bytes */
  Elf32_Word    sh_link;        /* Link to another section */
  Elf32_Word    sh_info;        /* Additional section information */
  Elf32_Word    sh_addralign;       /* Section alignment */
  Elf32_Word    sh_entsize;     /* Entry size if section holds table */
} Elf32_Shdr;
成员说明
sh_name节名称，是节区头字符串表节区中（Section Header String Table Section）的索引，因此该字段实际是一个数值。在字符串表中的具体内容是以 NULL 结尾的字符串
sh_type节的类型
sh_flags每一位代表不同的标志，说明节区是否可写、可执行等属性
sh_addr如果节区将出现在进程的内存映像中，此成员给出节区的第一个字节应该在进程镜像中的位置。否则，此字段为 0
sh_offset给出节区的第一个字节与文件开始处之间的偏移。
sh_size此成员给出节区的字节大小。
sh_link此成员给出节区头部表索引链接，其具体的解释依赖于节区类型
sh_info此成员给出附加信息，其解释依赖于节区类型
sh_addralign节区地址对齐大小. 例如，如果一个节区有一个 doubleword 类型的变量，那么系统必须保证整个节区按双字对齐
sh_entsize某些节区中存在具有固定大小的表项的表，如符号表. 对于这类节区，该成员给出每个表项的字节大小
  • sh_type定义的部分section类型如下
名称取值说明
SHT_NULL0无效段
SHT_PROGBITS1该类型节区包含程序定义的信息，它的格式和含义都由程序来决定. 程序段、代码段、数据段都是这种类型
SHT_SYMTAB2该段内容为符号表
SHT_STRTAB3该段的内容为字符串表
SHT_RELA4该段的内容为重定位表
SHT_HASH5符号表的Hash表
SHT_DYNAMIC6动态链接信息
SHT_NOTE7提示性信息
SHT_NOBITS8表示该段在文件中不占空间，比如.bss段
SHT_REL9该段包含了重定位信息
SHT_SHLIB10保留，语义尚未被定义
SHT_DYNSYM11动态链接的符号表

一个基本的ELF文件会包含三个最基本的常用section

名称说明
.text代码段

注意，显式存储的初始化静态变量也有可能在.bss段中，比如static int x = 0;

下面是一个简单的ELF文件结构及运行过程

弱符号和强符号​

对于C/C++来说，编译器默认函数和初始化了的全局变量为强符号，未初始化的全局变量为弱符号，也可以通过__attribute__((week))来将一个强符号转换为弱符号

对于强弱符号有以下规定：

  1. 不允许强符号多次定义
  2. 如果一个符号在某个目标文件中是强符号，在其他文件中都是弱符号，那么该符号是强符号
  3. 如果一个符号在所有目标文件中都是弱符号，那么选择其中占用空间最大的一个

关于-fno-stack-protector，如果直接使用gcc -c a.c b.c会报错：undefined reference to '__stack_chk_fail'

理由如下

编译源码到目标文件时，一定要加“-fno-stack-protector”，不然默认会调函数“stack_chk_fail”进行栈相关检查，然而是手动裸ld去链接，没有链接到“stack_chk_fail”所在库文件，所以在链接过程一定会报错: undefined reference to `__stack_chk_fail'。解决办法不是在链接过程中，而是在编译时加此参数，强制gcc不进行栈检查，从而解决。 此外，ld 的时候加上参数"-e main"就可以了，意思是将main函数作为程序入口，ld 的默认程序入口为_start。
