程序员的自我修养笔记(二) | 装载与动态链接

可执行文件的装载与进程​

虚拟地址​

每个程序运行起来之后具有自己独立的虚拟地址空间，虚拟地址空间大小由平台决定

比如32位虚拟地址空间为0x0000000~0xFFFFFFFF，即2^32 = 4GB；64位虚拟地址空间为0x0000000000000000~0xFFFFFFFFFFFFFFFF，即2^64 = 16 EB = 17,179,869,184 GB

Linux下32位将虚拟地址分成操作系统部分和用户部分：0x00000000~0xBFFFFFFF是用户部分，0xC0000000~0xFFFFFFFF是操作系统部分

装载​

比较常见的两种动态装载方式有：覆盖装入、页映射。覆盖装入现在基本被淘汰掉了

创建一个进程，装载相应的可执行文件并执行以下步骤

  • 创建一个独立的虚拟空间
  • 读取可执行文件，并且建立虚拟空间与可执行文件的映射关系
  • 将CPU指令寄存器设置成可执行文件的入口地址，开始运行

ELF文件装载进内存之后，与链接过程中的程序不同：运行时，操作系统会将相同类型的section合并成一段segment，以便于提高内存的利用率

比如现在有.init和.text段，前者是程序初始化的代码段，后者是用户编写的代码段。两者都是可执行的代码段，所以可以合并到一起，如果不合并到一起的话，操作系统会给这两个段分别分配页面，这样就造成了内存空间的浪费

动态链接​

为什么需要动态链接？​

  • 静态链接在多进程环境下会造成极大的内存空间浪费
  • 静态链接不利于软件更新，如果对静态链接库进行更新，还需要编写完重新链接再发布给用户

地址无关代码(PIC, Position-independent Code)​

如果有多个动态链接库需要被装载时，如何确定被装载的位置？

可执行文件基本可以确定自己在进程虚拟空间中的起始地址，因为其往往是第一个被加载的文件；但共享对象在编译时不能假设自己在进程虚拟地址空间中的位置。

我们需要通过装载时重定位来保证动态链接库装载时不会出错，GCC下通过添加参数-fPIC或者-shared来实现装载时重定位

我们这里用一个例子来说明常见代码地址无关如何实现

我们现在写了两个C代码

// main.c
static int a;
extern int b;

extern void ext();

void bar() {
    a = 1;
    b = 2;
}

void foo() {
    bar();
    ext();
}

int main() {
    foo();
    return 0;
}
// tool.c
int b;

void ext() {
    return ;
}

我们先编译动态链接库

gcc -shared -fPIC -m32 -o libmyshare.so tool.c

然后编译main.c

gcc main.c -m32 -o c.out -L. -lmyshare

模块内部的函数调用、跳转​

我们在foo()中调用了内部模块的bar()，让我们objdump -d c.out -M intel > objdump_test.txt看看汇编结果

模块内部的数据访问​

我们在bar()中访问了模块内部的static int a，看一下汇编

000011cd <bar>:
    11cd:   f3 0f 1e fb             endbr32 
    11d1:   55                      push   ebp
    11d2:   89 e5                   mov    ebp,esp
    11d4:   e8 65 00 00 00          call   123e <__x86.get_pc_thunk.ax>
    11d9:   05 fb 2d 00 00          add    eax,0x2dfb
    11de:   c7 80 38 00 00 00 01    mov    DWORD PTR [eax+0x38],0x1
    11e5:   00 00 00 
    11e8:   8b 80 18 00 00 00       mov    eax,DWORD PTR [eax+0x18]
    11ee:   c7 00 02 00 00 00       mov    DWORD PTR [eax],0x2
    11f4:   90                      nop
    11f5:   5d                      pop    ebp
    11f6:   c3                      ret

把函数调用所需要的栈帧操作去掉，得到了以下命令

; get a
    11d4:   e8 65 00 00 00          call   123e <__x86.get_pc_thunk.ax>
    11d9:   05 fb 2d 00 00          add    eax,0x2dfb
    11de:   c7 80 38 00 00 00 01    mov    DWORD PTR [eax+0x38],0x1
    11e5:   00 00 00

具体操作是这样的：

  1. 调用__x86.get_pc_thunk.ax，将call的下一条指令的地址存储到eax中
  2. eax += 0x2dfb
  3. a = *(eax+0x38) = 0x1

根据上面的操作，我们可以算出a的偏移地址是0x11d9+0x2dfb+0x38=0x400c

用gdb验证一下

可以看到基址为0x56555000，加上计算出的偏移地址就是a的地址

模块间调用、跳转​

动态链接中通过GOT表实现模块间函数调用、跳转

000011f7 <foo>:
    11f7:   f3 0f 1e fb             endbr32 
    11fb:   55                      push   ebp
    11fc:   89 e5                   mov    ebp,esp
    11fe:   53                      push   ebx
    11ff:   83 ec 04                sub    esp,0x4
    1202:   e8 c9 fe ff ff          call   10d0 <__x86.get_pc_thunk.bx>
    1207:   81 c3 cd 2d 00 00       add    ebx,0x2dcd
    120d:   e8 bb ff ff ff          call   11cd <bar>
    1212:   e8 69 fe ff ff          call   1080 <ext@plt>
    1217:   90                      nop
    1218:   83 c4 04                add    esp,0x4
    121b:   5b                      pop    ebx
    121c:   5d                      pop    ebp
    121d:   c3                      ret

在0x1212可以看到call 1080 <ext@plt>，通过延迟绑定的方式，将ext的地址放在GOT表中，等待链接时再重定位

延迟绑定​

延迟绑定用.plt和.got表来实现

我们以上面这个程序为例子，第一次调用ext()时，跳转到ext@plt，此时ebx=0x3fd4，这是.got的偏移地址

00001080 <ext@plt>:
    1080:   f3 0f 1e fb             endbr32 
    1084:   ff a3 10 00 00 00       jmp    DWORD PTR [ebx+0x10]
    108a:   66 0f 1f 44 00 00       nop    WORD PTR [eax+eax*1+0x0]

此时.got已经构造好了，所以直接跳转到.got中的地址，即ext()的地址

通常情况下，是如下的流程

ext@plt:
    jmp *(ext@GOT)
    push n
    push moduleID
    jmp __dl_runtime_resolve

第一条指令是跳转到.got中的地址，但是第一次执行的时候并不会跳转到相应的函数的位置，此时的地址指向下一条指令，即跳转到下一条指令（其实就是没有用）；第二条指令是将一个数字压栈，这个数字是ext这个符号引用在重定位表.rel.plt中的下标；第三条指令是压入模块ID；第四条指令是跳转到__dl_runtime_resolve函数，这个函数是动态链接器提供的，用来解析符号地址，填写到真正的.got表中

上面的操作完成后，.got表才被解析成功，下次调用ext()时，就可以直接跳转到ext()的地址了
