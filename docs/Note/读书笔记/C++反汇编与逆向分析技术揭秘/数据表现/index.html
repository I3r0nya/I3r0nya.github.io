基本的数据表现与表达式求值

const​

const 定义的常量本质上是一种假的常量，const 修饰的变量会在编译期间检查是否有修改的行为，从而保证常量不被修改。但是我们可以直接得到 const 修饰的变量的地址，通过指针修改变量的值，比如

#include <cstdio>

int main(int argc, char *argv[]) {
    const int n1 = 5;
    int *p = (int *)&n1;
    *p = 10;
    int n2 = n1;
    printf("%d\n", n2);
    return 0;
}

这一段代码可以成功通过编译，通过调试我们可以看到 n1 的值已经被修改了，但是输出是 5，这是因为编译器对程序做了常量传播的优化，在汇编里面可以明显地看到

常用的优化​

编译器在结束语义分析之后，会进行汇编代码的生成，这一阶段会进行编译优化，通过编译优化来提高程序的效率，下面两个是书上介绍的比较常见的两种优化

常量传播​

将编译期间可计算出结果的变量转换为常量，减少变量的使用，比如

#include <cstdio>

int main(int argc, char *argv[]) {
    int n = 1;
    printf("%d\n", n);
    return 0;
}

表达式求值​

除法​

计算机中的除法分为两种，一个是无符号/有符号整数除法，一个是浮点数除法

对于整数除法而言，总是会有除不尽的可能，即余数大于0，对于这种情况，不同的语言的规定不同。比如对于算式

x=−32x = \frac{-3}{2}x=2−3​

Python会对 xxx 向下取整，取在 −∞-\infty−∞ 方向最接近 xxx 的整数值

而C/C++则是选择对 xxx 向零取整，取往0方向最接近 xxx 的值

同样，对于求余/取模的行为，不同的语言也有不同的规定

求余/取模的过程是相似的，对于被除数 aaa ，除数 bbb ，商 qqq ， 余数 rrr

a=bq+rb=a−rqq=a−rbr=a−qb\begin{matrix} a & = bq + r \\ b & = \dfrac{a - r}{q} \\ q & = \dfrac{a - r}{b} \\ r & = a - qb \end{matrix}abqr​=bq+r=qa−r​=ba−r​=a−qb​

区别在于，求余遵循尽可能让商向0靠近，取模遵循尽可能让商向负无穷靠近。

还是用Python来举例，Wiki上介绍的比较清楚，% 是取模运算， math.fmod 是求余运算

static LIBDIVIDE_INLINE struct libdivide_u32_t libdivide_internal_u32_gen(
    uint32_t d, int branchfree) {
    if (d == 0) {
        LIBDIVIDE_ERROR("divider must be != 0");
    }

    struct libdivide_u32_t result;
    uint32_t floor_log_2_d = 31 - libdivide_count_leading_zeros32(d);

    // Power of 2
    if ((d & (d - 1)) == 0) {
        // We need to subtract 1 from the shift value in case of an unsigned
        // branchfree divider because there is a hardcoded right shift by 1
        // in its division algorithm. Because of this we also need to add back
        // 1 in its recovery algorithm.
        result.magic = 0;
        result.more = (uint8_t)(floor_log_2_d - (branchfree != 0));
    } else {
        uint8_t more;
        uint32_t rem, proposed_m;
        proposed_m = libdivide_64_div_32_to_32((uint32_t)1 << floor_log_2_d, 0, d, &rem); // 2^{32 + l} / d

        LIBDIVIDE_ASSERT(rem > 0 && rem < d);
        const uint32_t e = d - rem;

        // This power works if e < 2**floor_log_2_d.
        if (!branchfree && (e < ((uint32_t)1 << floor_log_2_d))) {
            // This power works
            more = (uint8_t)floor_log_2_d;
        } else {
            // We have to use the general 33-bit algorithm.  We need to compute
            // (2**power) / d. However, we already have (2**(power-1))/d and
            // its remainder.  By doubling both, and then correcting the
            // remainder, we can compute the larger division.
            // don't care about overflow here - in fact, we expect it
            proposed_m += proposed_m;
            const uint32_t twice_rem = rem + rem;
            if (twice_rem >= d || twice_rem < rem) proposed_m += 1;
            more = (uint8_t)(floor_log_2_d | LIBDIVIDE_ADD_MARKER); // LIBDIVIDE_ADD_MARKER = 0x40
        }
        result.magic = 1 + proposed_m;
        result.more = more;
        // result.more's shift should in general be ceil_log_2_d. But if we
        // used the smaller power, we subtract one from the shift because we're
        // using the smaller power. If we're using the larger power, we
        // subtract one from the shift because it's taken care of by the add
        // indicator. So floor_log_2_d happens to be correct in both cases.
    }
    return result;
}

struct libdivide_u32_t libdivide_u32_gen(uint32_t d) {
    return libdivide_internal_u32_gen(d, 0);
}

uint32_t libdivide_u32_do(uint32_t numer, const struct libdivide_u32_t *denom) {
    uint8_t more = denom->more;
    if (!denom->magic) {
        return numer >> more;
    } else {
        uint32_t q = libdivide_mullhi_u32(denom->magic, numer);
        if (more & LIBDIVIDE_ADD_MARKER) {
            uint32_t t = ((numer - q) >> 1) + q;
            return t >> (more & LIBDIVIDE_32_SHIFT_MASK); // LIBDIVIDE_32_SHIFT_MASK = 0x1F 
        } else {
            // All upper bits are 0,
            // don't need to mask them off.
            return q >> more;
        }
    }
}
