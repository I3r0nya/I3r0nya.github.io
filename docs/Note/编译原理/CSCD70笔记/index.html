Introduction​

编译器的结构​

代码优化的几个要素：

  • Formulate optimization problem (制定优化的问题)
  • Representation (抽象地表示优化相关的细节)
  • Analysis (分析)
  • Code Transformation (转换)
  • Experimental Evaluation

Basic Block (基本块)​

Basic Block是一段连续的最大中间指令序列，它必须满足下面几个条件：

  • 外部控制流只能从Basic Block的第一条指令进入
  • 除非在块末尾，否则Basic Block中所有的指令都会连续执行，不存在分支或停止

我们在Basic Block中进行的优化称为局部优化

备注

通常有Label的语句块就是一个Basic Block，因为它满足第一点，一般都会直接jmp <label>

Flow Graphs (控制流图)​

Flow Graphs是一张控制流图，它的相关定义如下：

  • 节点：Basic Block

  • 边：Bi→BjB_i \rightarrow B_jBi​→Bj​，通过BiB_iBi​块中的一些指令可以跳转到BjB_jBj​

    如何确定边？

    • 第一种情况，BiB_iBi​的结尾存在一个goto形式的跳转到BjB_jBj​的开头
    • 第二种情况，BjB_jBj​在物理上紧跟在BiB_iBi​的后面，且BiB_iBi​结尾不存在跳转指令
备注

如何划分Basic Block？

  • 我们需要标记出每个Basic Block的Leader，它通常是：
    • 第一条指令
    • 所有跳转指令指向的指令
    • 所有跳转指令之后的一条指令

每一个Basic Block从Leader开始，并在Leader或者最后一条指令之前结束

优化​

  • 局部优化

    • Common subexpression elimination (消除公共子表达式) 如果表达式 a op b 在之前已经计算过且在下一次调用前没有修改，我们称为公共子表达式，且这次计算就可以忽略掉
    • Constant folding or elimination (常量折叠、常量传播) 在编译期间计算表达式，并直接用结果替代表达式
    • Dead code elimination (删除无用代码) 删除永远不会被执行、被访问的代码

  • 全局优化

    • 全局版本的局部优化
    • Loop optimizations (循环优化)
      • 减少每次迭代中执行的代码 比如将循环中不管循环多少次结果都不会变的计算移动到循环外面以减少计算次数
    • 消除控制流上相同的代码段

  • 归纳变量消除

    对于一个变量x，如果存在一个正的或负的常量c，使得每次x被赋值时，它的值总是增加c，则称x为归纳变量。在沿着循环运行时，如果有一组归纳变量的值的变化保持步调一致，常常可以将这组变量删除为只剩一个。通常循环索引就是一个归纳变量

    • Strength reduction (强度削弱) 将乘法、除法替换成位运算
    • Elimination of loop index (变量替换)

Local Optimization - Local analysis​

DAG​

局部优化基本块我们需要先构建出基本块的有向无环图(DAG)

  • 基本块中每个语句s都对应一个节点N
  • 节点N的标号是s中的运算符，同时还有一个定值变量表和节点相关联，表示s是在此基本块中最晚对表中变量进行定值的语句
  • N的子节点是基本块中在s之前、最后一个对s所使用的运算分量进行定值得语句对应的节点。如果s的某个运算分量在基本块内没有在s之前被定值，那么这个运算分量对应的子节点就是代表该运算分量初始值的叶节点
  • 在为语句x = y + z构造节点N的时候，如果x已经在某个节点M的定值变量表中，则从M的定值变量表中删除变量x

Value Numbering​

我们通过构造一个动态映射表来储存表达式的值，当之后再次出现相同的表达式，可以直接通过查表来替换，否则插入到表中

一个比较简单的伪算法如下：

比如我们现在有这样一组指令：

a = b + c;
b = a - d;
c = b + c;
d = a - d;

用ValueNumbering构造出表：

Assign: a->r1, b->r2, c->r3, d->r4
a = b + c;    add t1 = r2, r3
              cpy r1 = t1
b = a - d;    sub t2 = r1, r4
              cpy r2 = t2
c = b + c;    add t3 = r2, r3
              cpy r3 = t3
d = a - d;    sub t4 = r1, r4
              cpy r4 = t4

我们发现d = a - d这条指令的两个数在之前并没有修改过，动态映射得到的结果是相同的，所以我们可以直接替换掉，相当于一次公共子表达式消除

Assign: a->r1, b->r2, c->r3, d->r4
a = b + c;    add t1 = r2, r3
              cpy r1 = t1
b = a - d;    sub t2 = r1, r4
              cpy r2 = t2
c = b + c;    add t3 = r2, r3
              cpy r3 = t3
d = a - d;    cpy r4 = t2  <- Optimization!

Global Optimization - DataFlow analysis​

数据流分析通过分析每一个基本块，从基本块中获取信息来进行优化

我们为每一个基本块定义两个集合 in(b)\text{in}(b)in(b) 和 out(b)\text{out}(b)out(b)：其中 in(b)\text{in}(b)in(b) 表示一个基本块的输入的数据流，out(b)\text{out}(b)out(b) 表示一个基本块输出的数据流，数据是基本块中的一条statement

比如对于一个基本块

d0: y = 3
d1: x = 10
d2: y = 11
    if e

d0、d1、d2就是数据流中的数据

数据流分析有三种分析方式

  • Reaching Definitions Analysis (到达-定值Forward分析)

    在Reaching-Definition分析中，我们首先需要对一条statement的影响进行分析，比如

    a = b + c

    这条statement使用了变量b、c，取代(Kill)了旧的定义，定义了新的a

    判断一条数据是否能到达某个节点，只需要判断是否存在一条路径能让这条数据到达这一点时未被替代

    我们用下面的式子来计算 out(b)\text{out}(b)out(b) 和 in(b)\text{in}(b)in(b)

    out(b)=fb(in(b))=Gen(b)∪(in(b)−Kill(b))in(b)=⋂P是b的前驱基本块(out(P))\begin{aligned} \text{out}(b) & = f_b(\text{in}(b)) = \text{Gen}(b) \cup (\text{in}(b) - \text{Kill}(b)) \\ \text{in}(b) & = \bigcap_{\text{P是b的前驱基本块}} (out(P)) \end{aligned}out(b)in(b)​=fb​(in(b))=Gen(b)∪(in(b)−Kill(b))=P是b的前驱基本块⋂​(out(P))​

    Gen(b)\text{Gen}(b)Gen(b) 集合代表Generates definitions，表示在基本块从开头到结尾未被重新定义的数据集合

    比如

    d0: y = 3
    d1: x = 10
    d2: y = 11

    d0在到达结尾之前，被d2重新定义了，所以d0这条数据就被Kill掉了，并不在 Gen(b)\text{Gen}(b)Gen(b) 里

    Kill(b)\text{Kill}(b)Kill(b) 表示所有基本块中被取代的数据集合

    备注

    这里的示例仅按照定义未进行优化，在 f1f_1f1​ 中的 Kill(1)\text{Kill}(1)Kill(1) 集合包括 d2d_2d2​ ，这是因为从全局的角度来看， d2d_2d2​ 被 d0d_0d0​ Kill掉了

  • Live Variable Analysis (实时-变量Backward分析)

    对于Live-Variable分析来说，我们需要定义一个变量是否是活跃的：

    如果变量v在某些路径中被使用了，则v是活跃的，否则就是非活跃的

    我们用下面的式子来计算 out(b)\text{out}(b)out(b) 和 in(b)\text{in}(b)in(b)

    in(b)=fb(out(b))=Use(b)∪(out(b)−Def(b))out(b)=⋃S是b的后继基本块(in(S))\begin{aligned} \text{in}(b) & = f_b(\text{out}(b)) = \text{Use}(b) \cup (\text{out}(b) - \text{Def}(b)) \\ \text{out}(b) & = \bigcup_{\text{S是b的后继基本块}} (in(S)) \end{aligned}in(b)out(b)​=fb​(out(b))=Use(b)∪(out(b)−Def(b))=S是b的后继基本块⋃​(in(S))​
    • Use(b)\text{Use}(b)Use(b) 表示在基本块b中被使用，但在使用前没有被定义的变量集合
    • Def(b)\text{Def}(b)Def(b) 表示在基本块b中被定义，但在定义前并没有被使用的变量集合

需要注意的是循环的迭代条件，Reaching-Definition需要对比前后结果的 out[b]out[b]out[b] ，如果不同则需要继续遍历；而Live-Variables对比的是 in[b]in[b]in[b]. 同时注意边界条件和初始化

  • Available-Expressions Analysis (可用表达式Forward分析)

    如果从CFG的首节点到达程序点p的每条路径都对表达式x op y进行计算，并且从最后一个这样的计算到点p之间没有再次对x或y进行定义，那么表达式x op y在点p是Available的

    对 in(b)in(b)in(b) 和 out(b)out(b)out(b) 的定义如下：

    out(b)=Gen(b)∪(in(b)−kill(b))in(b)=⋂P是b的前驱基本块(out(P))\begin{aligned} out(b) & = Gen(b) \cup (in(b) - kill(b)) \\ in(b) & = \bigcap_{\text{P是b的前驱基本块}} (out(P)) \end{aligned}out(b)in(b)​=Gen(b)∪(in(b)−kill(b))=P是b的前驱基本块⋂​(out(P))​
    • Gen(b)Gen(b)Gen(b) 表示基本块内对x op y进行了计算，且之后没有重新定义x或y的所有Expression的集合
    • Kill(b)Kill(b)Kill(b) 表示基本块内对x或y进行了计算，且之后没有重新计算x op y的所有Expression的集合

  • Iterative Algorithm for May & Forward Analysis

    对May & Forward Analysis我们可以抽象出一个元组模型

    对于CFG中所有节点，我m们定义一个k-元组 (out[n1],out[n2],…,out[nk])(out[n_1],out[n_2],\dots,out[n_k])(out[n1​],out[n2​],…,out[nk​]) 作为集合 (V1×V2⋯×Vk)(V_1 \times V_2 \dots \times V_k)(V1​×V2​⋯×Vk​) 的元素，表示为 VkV^kVk

    我们之前所做的迭代（传递函数 fsf_sfs​）相当于更新 VkV^kVk ，我们抽象成 F:Vk→VkF: V^k \rightarrow V^kF:Vk→Vk. 我们通过比较生成的 VkV^kVk 与上一次生成的 VkV^kVk，如果相同就停止，此时我们称达到了不动点(fixed point)

  • Math Foundation

    • 偏序集(Partial Order) 定义 (P,⊑)(P, \sqsubseteq)(P,⊑)，⊑\sqsubseteq⊑ 是一种关系来定义 PPP 中的元素，当 PPP 中元素满足：

      • ∀x∈P,x⊑x(Reflexivity)\forall x \in P, x \sqsubseteq x \quad \text{(Reflexivity)}∀x∈P,x⊑x(Reflexivity)
      • ∀x,y∈P,x⊑y∧y⊑x⇒x=y(Antisymmetry)\forall x, y \in P, x \sqsubseteq y \wedge y \sqsubseteq x \Rightarrow x = y \quad \text{(Antisymmetry)}∀x,y∈P,x⊑y∧y⊑x⇒x=y(Antisymmetry)
      • ∀x,y,z∈P,x⊑y∧y⊑z⇒x⊑z(Transitivity)\forall x, y, z \in P, x \sqsubseteq y \wedge y \sqsubseteq z \Rightarrow x \sqsubseteq z \quad \text{(Transitivity)}∀x,y,z∈P,x⊑y∧y⊑z⇒x⊑z(Transitivity)

      称 (P,⊑)(P, \sqsubseteq)(P,⊑) 为偏序集

    • 上下界(Upper and Lower Bounds) 给一个偏序集 (P,⊑)(P, \sqsubseteq)(P,⊑)， S⊂PS \subset PS⊂P，我们称

      • 对于 u∈Pu \in Pu∈P ，如果满足 ∀x∈S,x⊑u\forall x \in S, x \sqsubseteq u∀x∈S,x⊑u ，那么 uuu 是 SSS 的一个上界
      • 对于 ℓ∈P\ell \in Pℓ∈P ，如果满足 ∀x∈S,ℓ⊑x\forall x \in S, \ell \sqsubseteq x∀x∈S,ℓ⊑x ，那么 ℓ\ellℓ 是 SSS 的一个下界
      • 对于每一个 SSS 的上界如果存在一个上界 ⊔S\sqcup S⊔S ，满足 ⊔S⊑u\sqcup S \sqsubseteq u⊔S⊑u ，那么我们称 ⊔S\sqcup S⊔S 为最小上界(least upper bound, lub, join)
      • 对于每一个 SSS 的下界如果存在一个下界 ⊓S\sqcap S⊓S ，满足 \ell \sqsubseteq \sqcap S $ ，那么我们称 $\sqcap S 为最大下界(greatest lower bound, glb, meet)
      1. 不是所有偏序集都有最小上界和最大下界
      2. 如果偏序集有最小上界或最大下界，那么一定是唯一的
      3. 特殊的，如果 SSS 仅有两个元素 S=a,bS = {a, b}S=a,b，那么 ⊔S\sqcup S⊔S 可以表示为 a⊔ba \sqcup ba⊔b ；⊓S\sqcap S⊓S 可以表示为 a⊓ba \sqcap ba⊓b

    • 格(Lattice) 给定一个偏序集 (P,⊑)(P, \sqsubseteq)(P,⊑) ，∀a,b∈P\forall a, b \in P∀a,b∈P ，如果 a⊔b,a⊓ba \sqcup b, a \sqcap ba⊔b,a⊓b 都存在，那么偏序集 (P,⊑)(P, \sqsubseteq)(P,⊑) 是一个格(Lattice)

      • 如果只有一个存在，称为半格(Semi-Lattice)
      • 如果对于格的所有子集的 a⊔b,a⊓ba \sqcup b, a \sqcap ba⊔b,a⊓b 都存在，那么称为全格(Complete-Lattice)

        对于全格，都存在一个最大元素 ⊤=⊔P\top = \sqcup P⊤=⊔P ，称为Top；最小元素 ⊥=⊓P\bot = \sqcap P⊥=⊓P 称为Bottom

    • 不动点定理 给定一个全格 (L,⊑)(L, \sqsubseteq)(L,⊑) ，如果

      • f:L→Lf: L \rightarrow Lf:L→L 是单调的
      • LLL 是有穷的

      那么，通过 f(⊥),f(f(⊥)),…,fk(⊥)f(\bot), f(f(\bot)), \dots , f^k(\bot)f(⊥),f(f(⊥)),…,fk(⊥) 找到的第一个不动点为 最小不动点；通过 f(⊤),f(f(⊤)),…,fk(⊤)f(\top), f(f(\top)), \dots , f^k(\top)f(⊤),f(f(⊤)),…,fk(⊤) 找到的第一个不动点为 最大不动点

Assignment 1 Introduction to LLVM​

Assignment1的任务主要是熟悉如何编写简单的LLVM Pass，完成这个任务需要掌握CMake

  • 配置环境(ArchLinux)

    可以拉源码编译，也可以直接包管理安装

    sudo pacman -S llvm libc++ lldb

    因为pacman更新太快，所以使用的是llvm-14，比较大的区别是使用了不同的PassManager，11-12使用的是Legacy PM，13之后默认使用New PM，在使用opt的时候需要注意

  • 编写第一个LLVM Pass - FunctionInfo

    这里需要看完WritingAnLLVMPass

    第一个Pass我们需要实现统计Module的函数，包括

    • 函数名(Name)
    • 函数参数个数(Number of Arguments)
    • 函数在同一个Module中被直接调用的次数(Number of Direct Call Sites in the same LLVM module)
    • 函数内基本块的个数(Number of Basic Blocks)
    • 函数内指令的个数(Number of Instructions)
代码
#include "llvm/Passes/PassBuilder.h"
#include "llvm/Passes/PassPlugin.h"
#include "llvm/Support/raw_os_ostream.h"
#include <llvm/IR/PassManager.h>

using namespace llvm;

namespace {

void PrintFunctionInfo(Module &M) {
    outs() << "CSCD70 Functions Information Pass"
           << "\n";

    outs() << M.getName() << "\n";

    for (auto &F : M) {
        outs() << "Name: " << F.getName() << "\n";
        outs() << "Number of Arguments: " << F.arg_size() << "\n";
        outs() << "Number of Direct Call sites in the same LLVM module: "
               << F.getNumUses() << "\n";
        outs() << "Number of Basic Blocks: " << F.getBasicBlockList().size()
               << "\n";
        outs() << "Number of Instructions: " << F.getInstructionCount() << "\n";
    }
}

class FunctionInfo : public PassInfoMixin<FunctionInfo> {
    PreservedAnalyses run(Module &M, ModuleAnalysisManager &MAM) {
        PrintFunctionInfo(M);
        return PreservedAnalyses::all();
    }

    static bool isRequited() { return true; }
};

} // namespace

llvm::PassPluginLibraryInfo getFunctionInfoPluginInfo() {
    return {LLVM_PLUGIN_API_VERSION, "FunctionInfo", LLVM_VERSION_STRING,
            [](PassBuilder &PB) {
                PB.registerPipelineParsingCallback(
                    [](StringRef Name, ModulePassManager &MPM,
                       ArrayRef<PassBuilder::PipelineElement>) {
                        if (Name == "function-info") {
                            MPM.addPass(FunctionInfo());
                            return true;
                        }
                        return false;
                    });
            }};
}

extern "C" LLVM_ATTRIBUTE_WEAK ::llvm::PassPluginLibraryInfo
llvmGetPassPluginInfo() {
    return getFunctionInfoPluginInfo();
}
